# 意图识别 — 三是三不是

> 把模糊需求变成可执行的边界合同。

## 核心判断

### 1. 这是范围合同，不是分析报告

- "不是"比"是"更重要——说"不"才能防止范围蔓延
- ❌ 不是需求文档、不是技术方案——不要讨论实现细节

### 2. 同一抽象层级是硬约束

混层级的三是三不是毫无价值。不要把"用JWT"（实现）和"支持SSO"（功能）混在同一组。

**写完自检三个问题：**
- 把一条"是"改成"不是"，语法自然吗？不自然 = 层级乱
- 只做"三是"不做其他，核心意图能达成吗？不能 = 漏了东西
- 删掉某条"不是"，有人会觉得少了什么吗？不会 = 这条废的

### 3. "不是"必须是合理的人会误以为在范围内的

每写完一个"不是"，问自己：不写这条，一个正常工程师会不会默认包含它？不会 → 删掉。真正的废话："不是人工智能系统"、"不是一款游戏"。

### 4. 不确定时不强行归类

标记"待确认"比分类错误代价小。承认不确定性本身就是一种精确。

## 触发边界

### 正例
- "帮我做一下意图识别" / "这个需求帮我做三是三不是"
- "帮我界定一下范围" / "明确边界"

### 负例
| 请求 | 实际走 |
|------|--------|
| "帮我分析这个技术方案" | technical-analysis |
| "Review 这段代码" | code-review |
| "帮我写个登录接口" | coding-style |
| "这个架构怎么设计" | GoF-OOP |

### 临近混淆
| 请求 | 处理 |
|------|------|
| "帮我分析一下这个需求" | 追问：要技术分析还是意图澄清？ |
| "帮我理一下思路" | 追问：要什么输出？ |

## 工作方法

### 第一步：剥离手段，找到意图

用户说的往往是手段，不是目的。**多问一层为什么。**

| 用户说的（手段） | 真正意图（目的） |
|-----------------|-----------------|
| "我需要一个数据库" | 需要持久化和查询结构化数据 |
| "帮我写个API" | 外部系统需要访问某个功能 |
| "加个缓存" | 某个查询太慢了 |
| "做一套权限系统" | 不同角色能访问不同功能 |

如果用户输入太模糊，连意图都提取不出来，**用区分性澄清追问**：给二选一，不给开放式问题。

| ❌ | ✅ |
|----|----|
| "你能说清楚一点吗？" | "你要的是「只有管理员能用」还是「所有登录用户都能用」？" |
| "这个导出要怎么做？" | "导出是为了「用户自己看」还是「对接其他系统」？" |

原则：能用"是/否"或"A/B"回答的才是好澄清。

### 第二步：四要素校验

意图不完整时不要硬写三是三不是。先检查四要素：

| 要素 | 问自己 | 缺失表现 |
|------|--------|---------|
| **Actor**（谁） | 谁执行？用户？系统？ | "要做一个导出功能"——谁导出？ |
| **Action**（做什么） | 具体动作？CRUD哪个？ | "要做一个用户管理"——管什么？ |
| **Object**（对什么） | 作用对象？ | "要做一个查询"——查什么？ |
| **Imperative**（约束） | 必须还是可选？ | "最好支持导出"——边界模糊 |

**缺任何一个 → 追问补齐 → 再继续。**

### 第三步：提取细节意图（关键补充）

在核心意图基础上，从原话里萃取**用户没说但对执行有关键影响的细节**：

| 维度 | 问什么 | 例子（"怎么还没配送"） |
|------|--------|------------------------|
| **实体** | 原话里有订单号/商品名/人名吗？ | 订单号、商品名 |
| **情绪** | 语气是焦急/理性/抱怨/试探？ | 焦急 |
| **紧迫度** | 时间词、情绪词显示多急？ | 高（"还没"） |
| **隐含需求** | 用户没说但合理需要的服务？ | 联系骑手、催促配送 |

**约束**：
- 实体必须是原文字眼（"奔驰"不能改成"汽车"）
- 情绪没明显词就填"中性"，不要硬贴
- 隐含需求要具体（"查询配送状态"），不要空话（"希望更好"）

### 第四步：写三是三不是

**三是**：三条肯定，同一层级，每条可验证（做完没做完能判断），覆盖核心价值。

**三不是**：三条否定，同一层级，每条必须满足——
- 不做也不会影响核心价值，但做了就是范围蔓延
- 一个合理的人可能默认会做的（否则这条不是废的）
- 是独立的新边界，不是"是"的镜像（不要写"不是线下系统"来对应"是线上系统"）

如果某个边界确实模糊，放入"待确认"。如果模糊是因为缺信息，用**区分性澄清**（A/B二选一）追问。

## 输出格式

```
🎯 核心意图
[动词+对象，一句话说清要做什么]

📋 四要素
- Actor：[谁]
- Action：[做什么]
- Object：[对什么做]
- Imperative：[约束/边界]

🔍 细节意图
- 实体：[原文萃取的关键词，逗号分隔]
- 情绪：[焦急/理性/抱怨/试探/中性]
- 紧迫度：[高/中/低]
- 隐含需求：[用户没说但合理需要的服务]

✅ 三是（要做的）
1. [是什么] — [一句话解释]
2. [是什么] — [一句话解释]
3. [是什么] — [一句话解释]

❌ 三不是（不做的）
1. [不是什么] — [为什么不是]
2. [不是什么] — [为什么不是]
3. [不是什么] — [为什么不是]

⚠️ 待确认（如有）
- [模糊项]：[为什么模糊，需要什么信息才能判断]
```

## 常见错误

| 错误 | 正确做法 |
|------|---------|
| 把手段当意图（"建数据库"） | 多问一层为什么（"数据持久化"） |
| 层级混乱（功能+非功能混一起） | 先确定维度再下笔，用自检三问 |
| 不是太弱（"不是游戏"） | 换成"不包含后台管理界面" |
| 三不是是"是"的镜像（"不是线下"↔"是线上"） | 不是是独立边界，不是对立面 |
| 四要素不全硬写 | 缺要素先追问 |
| 模糊时给开放式问题 | 给A/B二选一 |
| 不确定时强行归类 | 标记"待确认" |
| 细节实体瞎改（"奔驰"→"汽车"） | 必须原文字眼 |
| 隐含需求空话（"希望更好"） | 要具体（"查询配送状态"） |
