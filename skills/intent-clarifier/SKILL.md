# 意图识别 — 三是三不是

> 把模糊需求变成可执行的边界合同。

## 核心判断（模型不知道的）

### 三是三不是是范围合同，不是分析报告

- ✅ 输出是**决策工具**：让所有人知道什么做、什么不做
- ✅ "不是"比"是"更重要——说"不"才能防止范围蔓延
- ❌ 不是需求文档、不是技术方案、不是架构设计
- ❌ 不要在这个阶段讨论实现细节

### 同一抽象层级是硬约束（Azure 一致性原则）

- ✅ 所有"是"和"不是"必须在**同一维度**。混层级的三是三不是毫无价值
- 例子：如果意图是"用户认证"，三是可以是"密码登录 / SSO / MFA"，三不是可以是"角色权限 / 审计日志 / 用户画像"——这些都在功能层面
- ❌ 不要把"用JWT"（实现）和"支持SSO"（功能）混在同一组
- **自检**：写完后问自己——"如果我要把一个'是'变成一个'不是'，它在语法上是否自然？"如果不自然，说明层级混乱

### "不是"必须是一个合理的人会以为在范围内的东西

- ✅ 如果你的"不是"没人会觉得在范围内，那这组三不是就是废的
- 测试方法：每写完一个"不是"，问自己"如果不写这条，一个正常工程师会不会默认包含它？"——如果答案是"不会"，删掉重写
- ❌ 真正的废话："不是一个人工智能系统"、"不是一款游戏"

### 不确定时不强行归类（置信度阈值原则）

- ✅ 从 NLP 意图识别学到的最重要一课：**标记"未知"比分类错误代价小**
- 如果某个边界确实模糊，放入"待确认"而不是硬塞进"是"或"不是"
- 待确认不是敷衍，是精准——承认不确定性本身就是一种精确

## 触发边界

### 正例（该加载）
- "帮我做一下意图识别"
- "这个需求帮我做三是三不是"
- "帮我界定一下范围"
- "这个需求的范围是什么不是什么"
- "帮我明确一下边界"

### 负例（不该加载）
| 请求 | 原因 |
|------|------|
| "帮我分析这个技术方案" | 走 technical-analysis |
| "Review 这段代码" | 走 code-review |
| "帮我写个登录接口" | 走 coding-style |
| "这个架构怎么设计" | 走 GoF-OOP |
| "帮我估算工期" | 不触发任何 Skill，直接拒绝——反臆造原则 |

### 临近混淆
| 请求 | 实际应该 |
|------|----------|
| "帮我分析一下这个需求" | 可能是 technical-analysis（要出技术方案），追问用户要的是"技术分析"还是"意图澄清" |
| "帮我理一下思路" | 太模糊，追问具体要什么输出 |

## STAR 前置：停—想—辨

> 来源：STAR Framework (Stop-Think-Analyze-Respond)，用于处理模糊意图。

在写任何输出之前，先过这三步：

### Stop：用户说的 ≠ 用户想的

不要直接接受用户的表述。用户说"我要一个数据库"，他不是真的想要一个数据库——他想要的是"持久化和查询数据的能力"。**多问一层为什么。**

| 用户说的（手段） | 真正意图（目的） |
|-----------------|-----------------|
| "我需要一个数据库" | 需要持久化和查询结构化数据 |
| "帮我写个API" | 外部系统需要访问某个功能 |
| "加个缓存" | 某个查询太慢了 |
| "做一套权限系统" | 不同角色的人能访问不同的功能 |

### Think：什么会受影响？

确定意图后，快速扫描影响面：
- 这个意图涉及几个系统/模块？
- 会影响哪些已有的东西？
- 有没有隐含的前置依赖？

**低风险**（单模块、无依赖）→ 直接出三是三不是  
**中风险**（多模块、有调用方）→ 出完后补充影响面提示  
**高风险**（抽象请求、范围广）→ 出完三是三不是后加确认门，等用户确认再继续

### Analyze：区分意图的粒度

同一个用户输入，可以切在不同粒度：

```
"做一个用户登录"
  ├── 粗粒度意图：用户身份认证
  ├── 中粒度意图：Web端密码登录 + Token管理  
  └── 细粒度意图：登录表单 → 后端验证 → JWT签发 → 前端存储
```

**默认选择中粒度**——粗了没用，细了限制实现。但如果用户明显在讨论细节，就选细粒度。

## 意图完整性校验：四要素

> 来源：IBM 需求工程 — 一个需求可执行的前提是包含 Actor + Action + Object + Imperative。

在写三是三不是之前，用这四要素检查意图是否完整：

| 要素 | 含义 | 缺失时的表现 |
|------|------|-------------|
| **Actor**（谁） | 谁执行这个动作？用户？系统？管理员？ | "要做一个导出功能"——谁导出？ |
| **Action**（做什么） | 具体动作是什么？CRUD 里的哪个？ | "要做一个用户管理"——管什么？ |
| **Object**（对什么做） | 动作作用的对象是什么？ | "要做一个查询"——查什么？ |
| **Imperative**（约束） | 边界条件是什么？shall/must/should？ | "最好支持导出"——必须还是可选？ |

**缺任何一个 → 意图不完整 → 不要强行写三是三不是，先追问。**

## 工作流程

### 1. 提取核心意图

从用户输入中剥掉所有实现细节、技术词汇、假设方案，找到**为什么要做这件事**。

**坑点**：不要把用户提出的方案当作意图。用户说的往往是手段，不是目的。

### 2. 四要素校验

用 Actor / Action / Object / Imperative 检查意图完整性。缺要素 → 追问 → 补齐后再继续。

### 3. 写三是

三条肯定陈述，在**同一抽象层级**，描述这个意图的核心范围。

规则：
- 每条可验证（做完没做完能判断）
- 每条是独立的维度，不重叠
- 覆盖意图的核心价值

### 4. 写三不是

三条否定陈述，明确**不做**什么。每条必须是"不做也不会影响核心价值，但做了就是范围蔓延"。

规则：
- 每条是一个合理的人可能默认会做的
- 每条与"是"在同一抽象层级
- 每条有明确的边界线

**坑点**：三不是不是需求文档的补集，是**最容易产生分歧的边界**。只列有争议的、需要明确排除的。

### 5. 一致性自检

写完后对三是三不是做三道自检：

1. **层级检查**：如果我把一条"是"改成"不是"，语法上自然吗？（不自然 = 层级混乱）
2. **价值检查**：如果只做"三是"不做其他任何东西，核心意图达成了吗？（没达成 = 漏了东西）
3. **边界检查**：三不是里有没有一条，删掉之后没人会觉得少了什么？（有 = 这条不是废的）

### 6. 模糊地带处理

如果有些东西确实很难判断是还是不是，单独标注为"待确认"。

**待确认不等于敷衍**。NLP 意图识别的核心教训——置信度低于阈值时标记"未知"比强行分类代价小。需求工程同理。

## 区分性澄清：二选一而非开放式

> 来源：Discriminative Clarification Framework — 最好的澄清问题是让答案在两个候选中最大化差异。

当意图模糊时，**不要问开放式问题**。给出两个互斥的选项：

| ❌ 模糊追问 | ✅ 区分性追问 |
|------------|-------------|
| "你能说清楚一点吗？" | "你要的是「只有管理员能用」还是「所有登录用户都能用」？" |
| "这个导出要怎么做？" | "导出是为了「用户自己看」还是「对接其他系统」？" |
| "权限要做到什么程度？" | "权限是「角色级别」（管理员/普通用户）还是「资源级别」（这个文档只能张三看）？" |

**原则**：如果你的澄清问题可以用"是/否"或"A/B"回答，它就是好问题。如果需要用户写一段话来回答，它就是坏问题。

## 输出格式

```
🎯 核心意图
[一句话，说清为什么要做这件事]

📋 四要素
- Actor：[谁]
- Action：[做什么]
- Object：[对什么做]
- Imperative：[约束/边界]

✅ 三是（要做的）
1. [是什么] — [一句话解释]
2. [是什么] — [一句话解释]
3. [是什么] — [一句话解释]

❌ 三不是（不做的）
1. [不是什么] — [为什么不是]
2. [不是什么] — [为什么不是]
3. [不是什么] — [为什么不是]

⚠️ 待确认（如有）
- [模糊项]：[为什么模糊，需要什么信息才能判断]
```

## 常见错误

| 错误 | 为什么错 | 正确做法 |
|------|---------|---------|
| 把手段当意图 | "建数据库"不是意图，"数据持久化"才是 | 多问一层为什么 |
| 三是三不是层级混乱 | "支持高并发"（非功能）和"用户登录"（功能）混在一起 | 先确定分析维度再下笔，用一致性自检 |
| 不是太弱 | "不是一个游戏"——没人会觉得是 | 换成"不包含后台管理界面" |
| 不是太多 | 列出七八条 | 三不是是强制函数，逼你想清楚最重要的三个 |
| 三不是是"对立面" | 不要写"不是线下系统"来对应"是线上系统" | 不是是独立的新边界，不是是的镜像 |
| 跳过意图直接写三是三不是 | 意图错了，后面全白写 | 先确认意图再展开 |
| 四要素不全硬写 | 不知道 Actor 是谁就开始分析 | 缺要素先追问 |
| 模糊时给开放式问题 | "你能说清楚一点吗？"——用户只会更困惑 | 给二选一的区分性澄清 |
| 不确定时强行归类 | 硬塞进"是"或"不是"，后面执行时才发现范围错了 | 标记"待确认"，承认不确定性 |
