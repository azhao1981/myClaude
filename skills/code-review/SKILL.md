# 审查标准 (Audit Standards)

## 1. 技术文档审查 (Doc Review)
**适用对象**: 技术方案、分析报告、架构设计文档。

### 审查清单
1. **幻觉与臆造**: 
   - 文档是否包含了未被要求的“工期估算”？ -> **标记为废话，建议删除**。
   - 是否使用了“短期/长期”这种模糊词汇而没有定义具体技术指标？ -> **标记为不严谨**。
2. **完整性检查 (五层分析)**:
   - 是否缺失了“数据结构”定义？
   - 是否忽略了“破坏性分析” (Breaking Changes)？
3. **决策质量**:
   - 方案评分 (A-C) 是否有充分的技术理由支持？

## 2. 代码审查 (Code Review)
**适用对象**: 源代码、Diff。

### 核心原则
1. **好品味 (Good Taste)**:
   - 消除边界情况。
   - 数据结构优于代码逻辑。
2. **KISS & Anti-Bloat**:
   - 缩进 > 3层 -> **垃圾**。
   - 包含解释"What"的注释 -> **垃圾**。
3. **Git 规范**:
   - 是否包含无关的格式修改？ -> **打回**。

## 输出格式
### Review 报告
- **【类型】** 📄 文档审查 / 💻 代码审查
- **【评分】** 🟢 通过 | 🟡 需修改 | 🔴 拒绝
- **【致命问题】** (直指核心缺陷，如：逻辑漏洞、臆造工期、过度设计)
- **【改进建议】** (具体的修改步骤)
