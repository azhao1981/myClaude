# Skill 写作 Anti-Patterns

## 1. 把 Skill 写成教学文档

```markdown
# ❌ 错误
这个 Skill 会教你：
1. 什么是 SOLID 原则
2. S 是单一职责...
3. O 是开闭原则...

# ✅ 正确
当你想重构臃肿代码时，这个 Skill 告诉你：
- 什么时候该拆（超过 500 行）
- 什么时候不拆（不到 100 行且不常改动）
- 命名红线（禁止 mfcOrder）
```

**原因**：模型知道 SOLID，写了是在交税而非增强判断。

## 2. 把通用流程当正文

```markdown
# ❌ 错误
请按照以下步骤执行：
1. 理解需求
2. 拆解任务
3. 执行代码
4. 检查结果

# ✅ 正确
你的任务是判断用户是否真的需要领域分层：
- 超过 300 行的混杂代码 → 是
- 问 SOLID 概念 → 否，这是另一个 Skill
```

**原因**：通用流程是废话，模型自己知道。

## 3. 描述写得太宽

```markdown
# ❌ 错误
描述：当用户想写代码时触发。

# ✅ 正确
描述：Load when 用户想把臃肿的业务代码重构成领域分层架构，
      且代码量超过 300 行或涉及多个业务实体。
      Not when 用户在问概念或做简单函数。
```

**原因**：太宽会误触发其他 Skill，局部影响整体。

## 4. 没有边界意识

```markdown
# ❌ 错误
名称：代码重构专家
描述：帮你重构任何代码

# ✅ 正确
名称：领域分层重构专家
描述：Load when 用户想把混杂业务代码分成 Domain 层和 Infrastructure 层。
      Not when 简单函数抽取、问概念、API 实现。
```

**原因**：边界模糊的 Skill 是上下文税的浪费。

## 5. 正文堆在一起

```markdown
# ❌ 错误
SKILL.md 里写了 2000 行，包括：
- 所有设计模式的解释
- 所有 SOLID 原则的教程
- 100 个例子

# ✅ 正确
SKILL.md 保持轻量（<500 行），重型内容放到：
- references/design-patterns.md  # 按需加载
- references/solid-explanation.md
- scripts/validate.py
```

**原因**：正文太长会让每次加载都很贵。

## 6. 写完就发布

```markdown
# ❌ 错误
写完 SKILL.md 就放到 skills/ 目录

# ✅ 正确
1. 在分支上大量迭代
2. 用 Hero Set 跑正向用例
3. 用 Evil Set 跑边界用例
4. 确认没有影响相邻 Skill
5. 带着行为证据发布
```

**原因**：Skill 一改可能影响整个系统，要谨慎。

## 7. 用命令级指导

```markdown
# ❌ 错误
1. git checkout -b new-branch
2. git cherry-pick <commit>
3. 解决冲突
4. git push

# ✅ 正确
将 commit cherry-pick 到干净分支，解决冲突时保留原意。
如果不能干净合入，就解释原因。
```

**原因**：命令级脆弱，环境一变就断。意图级稳健。

## 8. 只写正例，不写负例

```markdown
# ❌ 错误
评测集：
- 用户问 A → 加载
- 用户问 B → 加载
- 用户问 C → 加载

# ✅ 正确
评测集：
正例：
- 用户问 A → 加载

负例：
- 用户问 X → 不加载（这是 Y Skill 的活）
- 用户问 Y → 不加载（这是 Z Skill 的活）
```

**原因**：边界比功能更重要。定义清楚边界，才能防止乱加载。
