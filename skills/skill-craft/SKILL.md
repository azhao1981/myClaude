# Skill: Skill 锻造师

> 融合 Karpathy 编程原则 + Perplexity Skill 方法论

---

## 核心判断（Opinion 注入）

这些是模型自己不知道的判断边界。

### 关于 Skill 价值

- ✅ Skill 价值 = 模型判断增强 - 上下文消耗
- ✅ Skill 不是操作手册，是给模型补上它**缺失的判断边界**
- ❌ 不要写通用流程（模型知道"先理解需求"）
- ❌ 不要写概念解释（什么是 SOLID、什么是 TDD，模型知道）

### 关于路由风险

- ✅ **Skill 最大风险是乱加载，不是漏加载**——一旦乱加载，影响所有其他 Skill
- ✅ 描述写太宽 → 误触发；写太窄 → 漏加载
- ✅ 边界定义比内容重要：先定义"什么时候不该触发"，再写正文

### 关于正文内容

- ✅ 坑点比流程更值钱：写"看起来可以做但千万别做"
- ✅ 从失败里学到的经验最有价值
- ✅ 领域判断/工程判断/审美判断——告诉模型它不知道的

### 关于验证

- ✅ 没有评测集的 Skill 不应该发布
- ✅ Hero Request Set（正例）+ Evil Set（负例）是最低要求

## 评测集（路由边界）

### 正例（该加载）

- "帮我写一个代码审查的 skill"
- "我想做个 skill 自动处理 PR"
- "怎么开发一个自动化 workflow 的技能"
- "帮我重构一下这个 GoF-OOP skill"
- "这个 skill 老是误触发，怎么修"

### 负例（不该加载）

| 请求 | 原因 |
|------|------|
| "给我讲讲什么是 skill" | 只是了解概念，不是做 skill |
| "skill 和 plugin 有什么区别" | 概念学习，不是构建 |
| "帮我写段代码" | 普通 coding task |
| "怎么用 TDD 的方式写代码" | 这是 tdd-workflow 的活 |

### 临近混淆

| 请求 | 实际应该 |
|------|----------|
| "帮我做一个自动化的东西" | 太模糊，追问具体是 Skill 还是脚本 |
| "教我怎么写 prompt" | 可能是 prompt-design，不是 skill-craft |

## 构建参考（当模型需要流程时）

> 以下是参考流程，不是强制步骤。模型应根据自己的判断决定做到哪一步。

### 流程概览

```
意图 → 评测集 → 描述 → 正文 → 层级 → 验证
```

### 意图阶段（先问清楚）

**先问清楚，不要急着写**：

| 问题 | 为什么要问 |
|------|-------------|
| 这个 Skill 要解决什么问题？ | 定义目标 |
| 用户会怎么问这个需求？ | 确定触发 phrasing |
| 什么时候**不应该**触发它？ | 定义边界（最重要） |
| 成功的样子是什么？ | 定义成功标准 |

### 描述阶段（50 词以内）

| 要求 | 说明 |
|------|------|
| 以 `Load when...` 开头 | 明确触发条件 |
| 控制在 50 词以内 | 索引层每个词都是税 |
| 来自真实查询 | 不是想象的，是真实的 |

### 正文阶段（意图级 + 坑点）

**正文不是第一步写的**——先有评测和路由边界，最后才写正文。

#### 注入 Opinion

告诉模型它**不知道**的：

| Opinion 类型 | 例子 |
|-------------|------|
| 领域判断 | "这个任务里什么重要" |
| 工程判断 | "什么是错的，什么边界不能越" |
| 失败经验 | "模型自己容易误判的地方" |
| 审美判断 | "符合特定场景的偏好" |

#### 从命令级到意图级

| 命令级（❌） | 意图级（✅） |
|--------------|--------------|
| 列出 1-2-3 步骤 | 告诉目标是什么 |
| 写死环境假设 | 说明边界条件 |
| 写通用流程 | 模型已知的不写 |

#### 坑点比流程更值钱

| 应该写 | 不该写 |
|--------|--------|
| 看起来可以做但千万别做 | 正常情况怎么走（模型知道） |
| 真实运行里会坑人的地方 | 通用流程（模型已知） |
| 从失败里学到的经验 | 详细步骤（写死了） |

### 目录层级

```
skill/
├── scripts/        # 确定性逻辑（每次都一样→写成代码）
├── references/     # 重型知识（条件加载）
└── SKILL.md       # 主文件保持轻量
```

| 目录 | 用途 | 加载时机 |
|------|------|----------|
| `scripts/` | 可执行脚本 | 模型需要执行时 |
| `references/` | 大篇幅参考 | 按需加载，不是上来就读 |

### 验证阶段

**发布前必须验证**：

| 验证项 | 怎么验 |
|--------|--------|
| Hero Request Set | 代表性查询能否正确处理 |
| Evil Set | 边界情况会不会误触发 |
| 影响相邻 Skill | 修改后其他 Skill 有没有变差 |

## 常见错误诊断

| 错误写法 | 问题 | 正确写法 |
|----------|------|----------|
| 列出 SOLID 五个原则 | 模型已知 | 写什么时候该用/不该用 |
| 写"先理解需求再执行" | 废话 | 写用户实际上会怎么问 |
| 描述写 100 词 | 太宽 | 控制在 50 词以内 |
| 全部堆在主文件 | 浪费上下文 | 重型内容放到 references/ |
| 写完就发 | 没有验证 | 先跑 Hero Set + Evil Set |
| 把步骤名暴露在 description | 内部细节泄露 | description 只说触发条件 |
