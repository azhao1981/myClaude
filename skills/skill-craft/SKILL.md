# Skill: Skill 锻造师

> 融合 Karpathy 编程原则 + Perplexity Skill 方法论

---

## 核心原则

### Perplexity 方法论
Skill 不是操作手册，是给模型补上它**缺失的判断边界**。

### Karpathy 编程原则
1. **Think Before Coding**：不要假设，不要隐藏困惑，先明确假设和权衡
2. **Simplicity First**：最小代码解决问题，不做过度设计
3. **Surgical Changes**：精准修改，只改该改的
4. **Goal-Driven Execution**：定义可验证的成功标准

---

## Skill 构建六步法

### Step 0: 理解意图（Think Before Coding）

**先问清楚，不要急着写**：

| 问题 | 为什么要问 |
|------|-------------|
| 这个 Skill 要解决什么问题？ | 定义目标 |
| 用户会怎么问这个需求？ | 确定触发 phrasing |
| 什么时候**不应该**触发它？ | 定义边界（最重要） |
| 成功的样子是什么？ | 定义成功标准 |

**产出**：一份初步的评测集

### Step 1: 写评测，定义路由边界

**Step 0 的本质**：先定义路由边界，再写内容。

**评测用例来源**：

| 类型 | 例子 |
|------|------|
| 真实用户查询 | "帮我把这个代码重构一下" |
| 已知失败案例 | 之前类似 Skill 误触发的情况 |
| 临近领域混淆 | "你是怎么工作的" vs "帮我写代码" |

**关键洞察**：
> Skill 最大风险不是"不加载"，而是"乱加载"——一旦乱加载，会影响其他 Skill。

### Step 2: 写描述（Description）

**描述是路由触发器**。

| 要求 | 说明 |
|------|------|
| 以 `Load when...` 开头 | 明确触发条件 |
| 控制在 50 词以内 | 索引层每个词都是税 |
| 来自真实查询 | 不是想象的，是真实的 |

**原则**：
- 写太宽 → 误触发（影响其他 Skill）
- 写太窄 → 漏加载（该用时不用）

### Step 3: 写正文（意图级 + 坑点）

**正文不是第一步写的**——先有评测和路由边界，最后才写正文。

#### 3.1 注入 Opinion

告诉模型它**不知道**的：

| Opinion 类型 | 例子 |
|-------------|------|
| 领域判断 | "这个任务里什么重要" |
| 工程判断 | "什么是错的，什么边界不能越" |
| 失败经验 | "模型自己容易误判的地方" |
| 审美判断 | "符合特定场景的偏好" |

#### 3.2 从命令级到意图级

| 命令级（❌） | 意图级（✅） |
|--------------|--------------|
| 列出 1-2-3 步骤 | 告诉目标是什么 |
| 写死环境假设 | 说明边界条件 |
| 写通用流程 | 模型已知的不写 |

**例子**：

```markdown
# 命令级（错误）
1. 先理解需求
2. 再拆解步骤
3. 再执行
4. 再总结

# 意图级（正确）
告诉模型：
- 什么时候该拆，什么时候不拆
- 什么看起来可以但千万别做
- 失败时怎么解释
```

#### 3.3 坑点比流程更值钱

| 应该写 | 不该写 |
|--------|--------|
| 看起来可以做但千万别做 | 正常情况怎么走（模型知道） |
| 真实运行里会坑人的地方 | 通用流程（模型已知） |
| 从失败里学到的经验 | 详细步骤（写死了） |

### Step 4: 目录层级

```
skill/
├── scripts/        # 确定性逻辑（每次都一样→写成代码）
├── references/     # 重型知识（条件加载）
└── SKILL.md       # 主文件保持轻量
```

| 目录 | 用途 | 加载时机 |
|------|------|----------|
| `scripts/` | 可执行脚本 | 模型需要执行时 |
| `references/` | 大篇幅参考 | 按需加载，不是上来就读 |

### Step 5: 验证（Goal-Driven Execution）

**发布前必须验证**：

| 验证项 | 怎么验 |
|--------|--------|
| Hero Request Set | 代表性查询能否正确处理 |
| Evil Set | 边界情况会不会误触发 |
| 影响相邻 Skill | 修改后其他 Skill 有没有变差 |

## 上下文税判断

```markdown
这个 Skill 值得交税吗？

交税理由（注入 Opinion）：
✅ 领域判断（这个任务里什么重要）
✅ 工程判断（什么是错的，什么边界不能越）
✅ 失败经验（从失败里学到的）
✅ 坑点（看起来可以但千万别做）

浪费税项（模型已知）：
❌ 通用流程（先理解、再拆解）
❌ 概念解释（什么是 SOLID、什么是 TDD）
❌ 详细步骤（把每个命令都列出来）

结论：Skill 价值 = 模型判断增强 - 上下文消耗
```

## 常见错误诊断

| 错误写法 | 问题 | 正确写法 |
|----------|------|----------|
| 列出 SOLID 五个原则 | 模型已知 | 写什么时候该用/不该用 |
| 写"先理解需求再执行" | 废话 | 写用户实际上会怎么问 |
| 描述写 100 词 | 太宽 | 控制在 50 词以内 |
| 全部堆在主文件 | 浪费上下文 | 重型内容放到 references/ |
| 写完就发 | 没有验证 | 先跑 Hero Set + Evil Set |
