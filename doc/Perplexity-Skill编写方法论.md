# Perplexity Skill 编写方法论

> 核心观点：Skill 不是操作手册，而是给模型补上它缺失的判断边界

## 一、问题：为什么你写的 Skill 是垃圾？

### 常见错误
很多工程师写 Skill 时，会不自觉地写成"操作手册"：
- 把每个命令、每个步骤、每个参数都列出来
- 认为写得更详细 = 更专业

### 真实原因
这种写法源于**工程师习惯写说明文档**，但在 Skill 场景下完全错误：
- **目标不同**：不是教新人操作，而是给已经懂很多的模型补上缺失的判断边界
- **模型已知**：大模型已经知道通用流程，写进去增量小但**浪费上下文**
- **上下文是税**：如果 Skill 不能让模型做出更正确的判断，就是在白白征税

### 核心公式
```
详细 ≠ 专业
 Skill 价值 = 模型判断增强 - 上下文消耗
```

## 二、核心理念：注入 Opinion

Skill 需要注入四类内容：

| Opinion 类型 | 内容 |
|-------------|------|
| **领域判断** | 这个任务里什么重要 |
| **工程判断** | 什么是错的，什么边界不能越 |
| **失败经验** | 模型自己容易误判的地方 |
| **审美判断** | 符合特定场景的偏好 |

## 三、Skill 构建六步法

### Step 0：先写评测，定义路由边界

**本质**：先定义这个 Skill 应该解决什么问题，以及不应该解决什么问题

**评测用例来源**：
- 真实用户查询
- 已知失败案例
- **临近领域混淆**：看起来像但不该加载该 Skill 的请求

**关键洞察**：
> Skill 最大风险之一不是"不加载"，而是"乱加载"
> 一旦乱加载，会让无关内容进入上下文，影响其他 Skill，甚至把模型带到错误方向

**所以**：step 0 的本质是先定义路由边界，再写 Skill 内容

---

### Step 1：写描述（Description）

**定位**：描述是路由触发器

**写作要求**：
- 以 `Load when...` 开头
- 控制在 **50 词以内**
- 用户意图描述最好源自真实查询

**原则**：
- 名称和描述位于索引层，每次会话都要付上下文税
- **每个词必须服务于路由判断**
- 写太宽 → 误触发；写太窄 → 漏加载

## 四、正文写作：从命令级到意图级

### 错误写法（命令级）
```
1. git log
2. git checkout main
3. git checkout -b new-branch
4. git cherry-pick <commit>
```

### 正确写法（意图级）
```
将 commit cherry-pick 到一个干净分支上。
解决冲突时保留原意。
如果不能干净合入，就解释原因。
```

### 为什么意图级更优？

| 命令级 | 意图级 |
|--------|--------|
| 脆弱，环境变化就断 | 稳健，适应环境变化 |
| 限制模型灵活性 | 给模型判断空间 |
| 步骤写死 | 目标明确 |

**原因**：模型知道 git，但不知道你的目标和边界

## 五、目录层级结构

```
skill/
├── scripts/           # 确定性逻辑
│   └── parse_log.py   # 解析日志、检查格式、转换数据
├── references/        # 重型知识（条件加载）
│   └── api-errors.md  # API 错误码，非 200 时才读
├── assets/            # 模板和 Schema
│   └── output.json    # 固定输出结构，减少漂移
├── config.json        # 首次运行用户设置
│   └── default_channel
└── skill.md           # 主文件（轻量）
```

### 设计原则
- **scripts/**：每次都一样 → 写成代码，让模型直接调用
- **references/**：重型内容 → 条件加载，需要时再读
- **assets/**：固定结构 → 减少生成漂移
- **config.json**：用户偏好 → 首次问，保存复用

> 目录层级不是文件整理，而是设计模型的导航路径

## 六、迭代与发布

### Step 4：迭代

**在分支上做大量迭代**：
1. 从主分支没有 Skill 的状态开始
2. 跑 Hero Request Set（代表性查询集）
3. 跑 Evil Set（负面/边界测试集）
4. 最后提交包含评测集的完整修改

**评审重点**：
- 解决了哪些请求
- 新增/减少了哪些复利
- 有无误加载
- 有无影响相邻 Skill
- 有无整体性能回归

**警示**：
> 描述里的小词改动可能对路由产生巨大影响，甚至溢出影响其他 Skills
> Skill 描述的修改不能当文案优化，看它更像改分类器的判别边界

### Step 5：发布

**发布 = 带着行为证据上线**

含义：经过完整验证后，Skill 必须证明：
1. 解决了真实问题
2. 守住了路由边界
3. 上下文税物有所值

## 七、完整概念清单

### 路由与加载相关

| 概念 | 定义 |
|------|------|
| **上下文税 (Context Tax)** | 每个 Skill 占用上下文空间，必须证明价值才能上线 |
| **路由边界 (Routing Boundary)** | Skill 的触发条件，决定何时加载/不加载 |
| **触发边界** | 描述中定义的是否加载的判断条件 |
| **索引层** | 名称和描述所在的层，每次会话都要付上下文税 |
| **误触发** | 不该加载时加载了 |
| **漏加载** | 该加载时没加载 |
| **误加载 (Misloading)** | 加载了不该加载的 Skill，导致无关内容进入上下文 |
| **乱加载** | 不该加载时加载，或加载了错误的内容 |
| **溢出效应 (Spillover)** | 一个 Skill 的修改影响其他 Skill 的路由 |
| **相邻 Skill** | 可能被当前 Skill 影响的其他 Skill |

### 正文与指导相关

| 概念 | 定义 |
|------|------|
| **意图级指导 (Intent-level Guidance)** | 告诉模型目标而非命令，适应环境变化 |
| **命令级指导** | 列出具体命令步骤，脆弱且限制模型 |
| **通用流程** | 模型已知的标准流程，写入 Skill 增量小 |
| **坑点 (Pitfalls)** | 看起来可以做但千万别做的高风险点 |
| **Negative Examples** | 边界用例，告诉模型什么不该做 |
| **判断边界** | 模型缺失的、需要人工注入的决策信息 |

### Opinion 相关

| 概念 | 定义 |
|------|------|
| **领域判断** | 任务中什么重要 |
| **工程判断** | 什么是错的，什么边界不能越 |
| **审美判断** | 符合特定场景的偏好 |
| **失败经验** | 从失败中学到的教训 |
| **人的判断** | 需要压缩进上下文的人工经验 |

### 评测与验证相关

| 概念 | 定义 |
|------|------|
| **评测用例** | 用于验证 Skill 效果的测试案例 |
| **正例** | 应该加载 Skill 的请求 |
| **负例** | 不该加载 Skill 的请求 |
| **临近领域混淆** | 看起来像但不该加载 Skill 的请求 |
| **Hero Request Set** | 代表性查询集，用于验证 Skill 效果 |
| **Evil Set** | 负面/边界测试集，测试误加载情况 |
| **复利** | 评测用例积累的价值——定义边界比定义功能更重要 |
| **性能回归** | Skill 修改后整体性能下降 |
| **行为证据** | 证明 Skill 解决了真实问题、守住路由边界的验证结果 |

### 内容结构相关

| 概念 | 定义 |
|------|------|
| **正文 (Body)** | Skill 的主体内容，不是第一步写的 |
| **条件加载** | 按需加载重型内容，而非一开始就塞入上下文 |
| **条件性内容** | 非必读、可按条件加载的内容 |
| **确定性逻辑/确定性脚本** | 每次都一样、可写成代码的逻辑 |
| **重型知识** | 大篇幅的参考资料，按需加载 |
| **生成漂移** | 模型生成内容偏离预期结构 |
| **固定结构** | 通过模板/Schema 约束输出格式 |
| **导航路径** | 目录层级设计的模型阅读路径 |
| **首次运行用户设置** | config.json 中存储的默认配置 |

### 目录组件

| 概念 | 定义 |
|------|------|
| **scripts/** | 确定性逻辑目录，存放可执行脚本 |
| **references/** | 重型知识目录，条件加载 |
| **assets/** | 模板和 Schema 目录 |
| **config.json** | 用户设置文件 |
| **skill.md** | 主文件，应保持轻量 |

### 流程与协作相关

| 概念 | 定义 |
|------|------|
| **上下文债务** | Skill 维护后变脏变长，成为负担 |
| **上下文消耗** | Skill 占用上下文的空间成本 |
| **分支迭代** | 在独立分支上验证 Skill 的方式 |
| **描述 (Description)** | 路由触发器，最难写的一行 |
| **上下文资源** | 会话中可用于推理的有限空间 |
| **操作手册** | 列出每个步骤的文档写法，Skill 应该避免 |
| **说明书** | 通用流程文档，模型已知，写入无意义 |
| **walk through** | 人类新人学习时的文档阅读方式 |
| **智能体系统** | Skill 运行的系统上下文，需考虑局部影响整体 |
| **通用工具** | 模型已知的标准工具，重复写浪费 token |
| **token** | 上下文消耗的单位，Skill 写作要避免浪费 |
| **分类器** | 描述承担的角色，修改描述如同改分类器边界 |
| **第一层问题** | Skill 上线前需回答的核心问题——上下文税是否值得 |
| **上下文空间** | 模型推理的有限资源，Skill 占用它必须证明价值 |
| **局部影响整体** | 每增加一个 Skill 都可能让其他 Skill 变差 |
| **维护** | Skill 上线后持续迭代更新的过程 |
| **路由上下文** | Skill 加载后对整体上下文的影响 |

### 流程与协作相关

| 对比概念 | 说明 |
|----------|------|
| 讲给 LLM vs 讲给同事 | 指导方式不同，对模型要跳过他已知的内容 |
| 内容 vs 税 | 判断标准：不能增强判断就是在交税 |
| 完整 vs 专业 | 质量标准：详细不等于专业，专业 = 增强模型判断 |
| 正常情况 vs 看起来可以但千万别做 | 内容优先级：坑点比流程更值钱 |

## 八、总结

### Skill 价值判断

```
if Skill 仅仅是流程罗列:
    不值得交税  # 模型自己知道

if Skill 能:
    - 证明解决了真实问题
    - 守住了路由边界
    - 把人的判断压缩进上下文
then:
    这笔税才有理由交
```

- [ ] Step 0：写了评测吗？定义路由边界了吗？

- [ ] Step 2：正文是意图级吗？写了坑点吗？
- [ ] Step 3：重型内容条件加载了吗？
- [ ] Step 4：在分支上迭代验证了吗？
- [ ] Step 5：有行为证据才发布了吗？
